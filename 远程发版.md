⚠️ **release.jks 一旦全部丢失，所有已装机的用户必须卸载重装才能继续升级**——千万别让它都消失。

2. iCloud Drive：`~/Library/Mobile Documents/com~apple~CloudDocs/XingShuHelper-keystore/`
3. OneDrive：`~/Library/CloudStorage/OneDrive-个人/XingShuHelper-keystore/`
4. GitHub Actions Secrets：`KEYSTORE_BASE64`（在仓库 Settings → Secrets 里）

**密码在哪看**：上面任意一处的 `CREDENTIALS.txt`。三个值（KEYSTORE_PASSWORD / KEY_ALIAS / KEY_PASSWORD），其中 alias 固定是 `xingshu`。

## GitHub Actions Secrets（已配置完毕）

仓库 → Settings → Secrets and variables → Actions：

| Secret 名 | 内容 | 来源 |
|---|---|---|
| `KEYSTORE_BASE64` | `release.jks` 的 base64 | `base64 -i release.jks` |
| `KEYSTORE_PASSWORD` | keystore 密码 | `CREDENTIALS.txt` |
| `KEY_ALIAS` | `xingshu` | 固定值 |
| `KEY_PASSWORD` | key 密码（与 KEYSTORE_PASSWORD 相同）| `CREDENTIALS.txt` |
| `DASHSCOPE_API_KEY` | 阿里云百炼 key | 同 `local.properties` 里的 |

**用 gh CLI 重置某个 secret**：
```bash
gh secret set KEY_NAME --repo yhLang/XingShuHelper --body "value"
```

## 日常发版流程（最常用）

```bash
# 1. 改完代码，先 commit + push
git commit -am "feat: ..."
git push origin main

# 2. 升版本号（语义化版本：bug 修复改 patch、新功能改 minor）
git tag v1.0.1
git push origin v1.0.1
```

3-5 分钟后：
- GitHub Actions 自动 build 签名 APK 并发到 Releases
- 你爸打开 App 主页 → 横幅显示"发现新版本 v1.0.1"→ 点"下载并安装"

**版本号规则**：
- `versionName`：从 tag 取（去掉 `v` 前缀），如 `v1.0.1` → `1.0.1`
- `versionCode`：CI 用 `git rev-list --count HEAD`（commit 数）自动递增
- 永远只升不降

**手动触发**（不打 tag，用现有版本号）：
GitHub 仓库 → Actions → Release APK → Run workflow → 输入版本号。

## 应用内自动更新流程

代码：[app/src/main/java/com/xingshu/helper/update/](app/src/main/java/com/xingshu/helper/update/)

- `UpdateChecker.kt` — 启动时调 `https://api.github.com/repos/yhLang/XingShuHelper/releases/latest`，比对 `BuildConfig.VERSION_NAME`，新版本则下载 APK。
- `UpdateBanner.kt` — 主页横幅 UI，显示更新提示 / 下载进度 / 拉起安装器。
- 安装走 FileProvider（[`app/src/main/res/xml/file_paths.xml`](app/src/main/res/xml/file_paths.xml)）+ `Intent.ACTION_VIEW` mime `application/vnd.android.package-archive`。
- 权限：`REQUEST_INSTALL_PACKAGES`（已在 manifest 里）。

要让某台手机能收到更新提示，前提是这台手机上装的 APK 版本里**已经包含上面这套 UpdateChecker 逻辑**（v1.0.0 起开始有）。早于 v1.0.0 的旧 APK 没有自检，必须手动装一次新版才能跟上自动通道。

## 常见问题处理

### CI 跑失败了

```bash
# 看最近一次失败原因
gh run list --repo yhLang/XingShuHelper --limit 3
gh run view <RUN_ID> --repo yhLang/XingShuHelper --log-failed | tail -80
```

**已踩过的坑（已解决，不会再发生）**：
- gradlew / gradle-wrapper.jar 没提交到仓库 → 已 commit。
- `*.jks` 在 `.gitignore` 里是对的，绝对不要把 keystore 提交到仓库。

### 想重置某个版本（重新 build 同 tag）

```bash
git tag -d v1.0.1
git push origin :refs/tags/v1.0.1
git tag v1.0.1
git push origin v1.0.1
```
注意：要把 GitHub 上的旧 Release 也手动删一下（CI 不会自动覆盖）：
```bash
gh release delete v1.0.1 --repo yhLang/XingShuHelper --yes
```

### 在新电脑上接手开发

```bash
gh repo clone yhLang/XingShuHelper
cd XingShuHelper
# 创建 local.properties（不在仓库里）
cat > local.properties <<EOF
sdk.dir=/Users/<你>/Library/Android/sdk
DASHSCOPE_API_KEY=sk-xxxxx
GITHUB_OWNER=yhLang
GITHUB_REPO=XingShuHelper
EOF
```
本地 debug build 不需要 keystore；只有 release build 需要，但 release 都走 CI 不在本地做。

### 仓库要改成 Private

可以，但 Releases assets 也会变私有，应用内更新会 401。两条路：
1. 单独建个 public 仓库专门发 release（CI 里把 release upload 改到那个仓库）。
2. 在 `UpdateChecker.kt` OkHttp 请求里加 `Authorization: token <PAT>`，PAT 通过 BuildConfig 注入。安全性较差（APK 反编译能拿到 PAT），需用 fine-grained PAT 限定单仓库只读。

### Keystore 真丢了怎么办

如果 4 处备份都没了，且 `KEYSTORE_BASE64` secret 也被删了——只能：
1. 用新 keystore 生成新 APK
2. 让用户卸载旧 App、重装新 App（数据不会自动迁移）

所以备份很重要。**绝对不要在任何时候执行 `gh secret delete KEYSTORE_BASE64`**。
